== Claim 1: Post-Quantum Cryptography (Kyber, Dilithium, SPHINCS+) with NIST PQC Winners

From README:
____
Post-Quantum Cryptography: Kyber (KEM), Dilithium (Digital signatures), SPHINCS+ (Hash-based signatures).
____

**How it's proven:**
- **Kyber export**: `KyberPublicKey`, `KyberSecretKey`, `kyber_keygen`, `kyber_encapsulate`, `kyber_decapsulate` at lines 74-77 in `src/ProvenCrypto.jl`
- **Dilithium export**: `DilithiumPublicKey`, `DilithiumSecretKey`, `dilithium_keygen`, `dilithium_sign`, `dilithium_verify` at lines 78-82
- **SPHINCS+ export**: `SPHINCSPublicKey`, `SPHINCSSecretKey`, `sphincs_keygen`, `sphincs_sign`, `sphincs_verify` at lines 83-87
- **Usage example**: README lines 75-90 show complete usage workflows for all three algorithms
- **Critical path**: Each algorithm implemented as pure Julia with deterministic output (e.g., `kyber_keygen(768)` for 192-bit equivalent security)

**Caveat:** Implementations are reference/educational grade, NOT production-ready. README explicitly states (line 148-157): "This library is for research and educational purposes... Not FIPS-Certified... NOT FIPS 140-2/3 certified. For compliance-critical systems, use FIPS-certified libraries via FFI."

== Claim 2: Hardware Acceleration Support (GPU, NPU, TPU, FPGA, DSP)

From README:
____
Hardware Acceleration: GPU (CUDA, ROCm, Metal), NPU/TPU, Crypto Instructions (AES-NI, SHA), Secure Enclaves (Intel SGX, AMD SEV, ARM TrustZone).
____

**How it's proven:**
- **Hardware detection**: `detect_hardware()`, `detect_simd_level()`, `detect_hardware_features()` exported at lines 61-64
- **Backend interface**: `AbstractCryptoBackend`, `CPUBackend` exported at lines 59-60
- **Modular strategy**: Each primitive (Kyber, Dilithium) can use different backends for NTT (Number Theoretic Transform) computation
- **Example output**: README lines 133-142 shows actual detection output: "MetalBackend(M3 + Neural Engine)", "CUDABackend(device=0, CC=8.9)", "CPUBackend(avx512, 16 threads)"
- **Critical path**: Operations auto-select best available backend at runtime; falls back to CPU-only if specialized hardware unavailable

**Caveat:** Actual backend implementations (CUDA, ROCm, Metal) are architecture stubs. The detection mechanism works, but specialized execution paths are not yet wired. All computations currently fall back to pure Julia CPU implementation.

== Dogfooded Across The Account

ProvenCrypto.jl is the **cryptographic foundation** for:
- **burble**: Uses ProvenCrypto for Noise Protocol authenticated channels and threshold cryptography in SIP media encryption
- **proven**: Rust equivalent that imports ProvenCrypto specs as reference (Layer 1 primitives via FFI, Layer 3 algorithms via formal proofs)
- **PolyglotFormalisms.jl**: Uses ProvenCrypto primitives for arithmetic property verification (mod arithmetic, random oracle properties)

Same cryptographic abstraction pattern (Layer 1/2/3) used in `gossamer` (IPC encryption) and `svalinn` (container security policies).

== Critical Invariants (What Must Not Break)

1. All FFI calls (libsodium, BoringSSL) must validate external library presence at module load time
2. Nonce uniqueness must be enforced: `aead_encrypt` must reject repeated (key, nonce) pairs
3. Kyber decapsulation must always return same shared secret as encapsulation (proved invariant in comment line 202)
4. Hardware detection must not crash on unsupported platformsâ€”must gracefully fall back to CPU
5. Pure Julia implementations must remain side-effect-free and deterministic
